# M1.1 Mac Host 动态原生档

## 用户原话

> 你是 Mac 动态原生档 Agent。目标：Mac 读取 Android 上报能力，生成 detected-native / detected-native-safe，不硬编码。

## 本轮目标

让 Mac Host 不再硬编码“原生画质”，而是根据 Android `HELLO.display_capabilities` 动态选择输出档位：

- `balanced`
- `hd60`
- `detected-native-safe`
- `detected-native`

## 明确不做

- 不修改 Android 代码。
- 不擅自修改 `设计/protocol-v0.md`；如发现字段缺失，只在报告中写建议。
- 不替换 TCP server 为双栈 POSIX socket（仍在 Network.framework 范围内）。
- 不进入虚拟显示、输入回传、Wi-Fi 发现等后续阶段。

## 已确认事实

- M1 冻结：Swift + ScreenCaptureKit + VideoToolbox H.264 + TCP 长连接。
- protocol v0 当前 HELLO 尚未定义 `display_capabilities`，本任务在 Mac 端先实现解析，协议文档补充交给 Protocol Agent 或主 Agent。
- 当前 Mac Host TCP server 为 IPv6-only，但不影响 profile 选择逻辑自测。
- 目标设备小米平板 6 Pro 常见横屏模式：2560x1600@144、1920x1200@60 等。

## 实现要求

1. Mac Host 建立 TCP 连接后读取 Android `HELLO`：
   - 如果收到 `display_capabilities`，解析并记录。
   - 如果没有收到或解析失败，降级到 `hd60` 或 `balanced`。
   - 不能因为缺少能力字段直接崩溃。
2. 实现 profile 选择：
   - `balanced`: 1280x800@30。
   - `hd60`: 1920x1200@60。
   - `detected-native-safe`: 使用 Android current mode 分辨率，FPS 上限 60。
   - `detected-native`: 使用 Android current mode 分辨率与刷新率，允许因编码器能力降级。
3. 码率估算不能写死：
   - 基准按 `width * height * fps * factor` 估算。
   - 设置上下限，避免离谱码率。
   - 实际公式写进报告。
4. SwiftUI App 显示：
   - Android 上报的 current mode。
   - selected profile。
   - 实际输出分辨率/FPS/码率。
   - 如果发生降级，显示降级原因。
5. CLI 支持：
   - `--profile detected-native-safe`
   - `--profile detected-native`
   - 可选 `--hello-fixture <path>` 用于无 Android 时自测 profile 选择逻辑。

## 验收标准

- [ ] `swift build` 编译通过（CLI + App）。
- [ ] CLI `--profile balanced --dump ...` 输出 1280x800@30。
- [ ] CLI `--profile hd60 --dump ...` 输出 1920x1200@60。
- [ ] CLI `--profile detected-native-safe --hello-fixture ... --dump ...` 按 fixture 的 current_mode 分辨率、60fps 上限输出。
- [ ] CLI `--profile detected-native --hello-fixture ... --dump ...` 按 fixture 的 current_mode 分辨率与刷新率输出。
- [ ] 无 fixture 的 `detected-native*` 明确降级并打印原因。
- [ ] TCP 模式能在收到 HELLO 后按 profile 开始推流；无 HELLO 时超时降级。
- [ ] SwiftUI App 显示 current mode、selected profile、输出配置、降级原因。
