## 动态原生档选择结果

Mac Host 优先读取 `current_mode.physical_width` / `physical_height`；`width` / `height` 仅作为旧 fixture 的兼容 fallback。`refresh_rate` 按容差归一化后再参与档位选择。

| profile | fixture current_mode | raw | normalized | selected | 选定输出 | 降级原因 |
|---|---|---|---|---|---|---|
| `balanced` | 任意 | - | - | - | 1280x800 @ 30fps, 2 Mbps | 无 |
| `hd60` | 任意 | - | - | - | 1920x1200 @ 60fps, 11 Mbps | 无 |
| `detected-native-safe` | 2880x1800@144 | 144.0 | 144 | 60 | 2880x1800 @ 60fps, 24 Mbps | raw=144.0Hz normalized=144Hz 超过 60fps，已限制到 60Hz |
| `detected-native` | 2880x1800@144 | 144.0 | 144 | 144 | 2880x1800 @ 144fps, 40 Mbps（上限） | raw=144.0Hz normalized=144Hz 超过编码器推荐上限 60Hz，允许运行但可能掉帧 |
| `detected-native-safe` | 无 fixture / 无 HELLO | - | - | 60 | 1920x1200 @ 60fps, 11 Mbps | 未收到 Android HELLO/display_capabilities，降级到 hd60 |
| `detected-native` | 无 fixture / 无 HELLO | - | - | 60 | 1920x1200 @ 60fps, 11 Mbps | 未收到 Android HELLO/display_capabilities，降级到 hd60 |
| `detected-native-safe` | 旧 fixture（仅 width/height） | - | - | 60 | 按 width/height 计算 | legacy compatibility: 未上报 physical_width/physical_height，使用旧 fixture 字段 width/height |
| `detected-native` | 1920x1200@59.94 | 59.94 | 60 | 60 | 1920x1200 @ 60fps, 11 Mbps | 无 |

## 刷新率容差归一化

`refresh_rate` 是浮点能力值，不允许用 `== 60` / `== 120` / `== 144` 做严格判断。Mac Host 按以下容差归一化：

```text
59.0 <= x <= 61.0   -> 60
89.0 <= x <= 91.0   -> 90
119.0 <= x <= 121.0 -> 120
143.0 <= x <= 145.0 -> 144
其他值              -> round(x)
```

选择逻辑：

- `detected-native-safe`：`selected_fps = min(normalized_fps, 60)`。
- `detected-native`：`selected_fps = normalized_fps`，但仍允许因编码器能力降级。

日志同时输出 `raw_refresh_rate`、`normalized_fps`、`selected_fps`。

## 码率估算规则

```text
factor = 0.08  (bits per pixel per frame)
raw_bitrate = width * height * fps * factor
bitrate = clamp(raw_bitrate, 2_000_000, 40_000_000)  // bps
```

示例：

- 1280x800@30 → raw = 2.46 Mbps → clamp 到 2 Mbps（下限）
- 1920x1200@60 → raw = 11.06 Mbps → 11 Mbps
- 2880x1800@60 → raw = 24.88 Mbps → 25 Mbps
- 2880x1800@144 → raw = 59.72 Mbps → clamp 到 40 Mbps（上限）

## 降级策略

1. `detected-native*` 未收到 HELLO 或 `display_capabilities` 解析失败 → 降级到 `hd60`。
2. `refresh_rate` 先按容差归一化，禁止用 `== 60 / == 120 / == 144` 严格判断。
3. `detected-native-safe`：`selected_fps = min(normalized_fps, 60)`；若 `normalized_fps > 60` → 记录限制原因。
4. `detected-native`：`selected_fps = normalized_fps`；若 `selected_fps > 60` → 标记可能掉帧警告。
5. TCP 连接后 5 秒内未收到 HELLO → 按无能力信息降级并继续推流。
6. 流已启动后收到迟到 HELLO → 忽略，避免编码器/采集器状态混乱。
7. 旧 fixture 未提供 `physical_width/physical_height` → 使用 `width/height` 并标记 `legacy compatibility`。

## 协议对齐

Mac Host 以 `设计/protocol-v0.md` 为准解析 HELLO：`display_capabilities.current_mode` 使用 `mode_id` / `physical_width` / `physical_height` / `refresh_rate`；`supported_modes` 内同样使用这些字段；`window_bounds`、`surface_size`、`density`、`density_dpi`、`orientation` 作为可选字段解析。`width` / `height` 仅保留为旧 fixture 的兼容 fallback。

- 长时间运行稳定性与发热。

## 发现的问题

1. TCP server 仍为 Network.framework IPv6-only；若 Android 真机通过局域网 IPv4 连接，需替换为双栈 POSIX socket server。
2. Network.framework 创建的 `NWListener` 在进程被 `SIGTERM` 终止后端口释放较慢，测试时需使用 `SIGKILL` 或等待；对日常双击打开 App 无影响。
3. `DisplayMode` 保留 `width` / `height` 作为旧 fixture 的 `legacy compatibility` fallback；新协议 payload 应始终使用 `physical_width` / `physical_height`。

## 需要主 Agent 合并或确认的事项

1. 是否在当前阶段就将 TCP server 替换为 POSIX socket 双栈实现。
2. `custom` profile 的命名与暴露方式（当前为内部兼容档，CLI 不展示为可选 `--profile custom`）。
