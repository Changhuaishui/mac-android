2. 明确 `display_capabilities` 的字段含义：
   - `current_mode` / `supported_modes` 来自 Android 运行时 `Display.Mode`。
   - `physical_width` / `physical_height` 是 Android Display Mode 物理模式尺寸，不是网页规格。

   - Android 发送该字段后不能要求旧 Mac Host 理解或回显。

## 协议兼容性说明

`display_capabilities` 只扩展 `HELLO` 的 JSON payload，不改变：

- TCP 传输方式。
- 32 字节 header。
- `magic` / `version` / `type`。
- `HELLO` 的方向和 flags。
- H.264 Annex B payload 约定。

因此旧 Android Client 不发送该字段仍合法；新 Android Client 发送该字段时，旧 Mac Host 只要忽略未知 JSON 字段即可继续工作。新 Mac Host 如果没有拿到该字段，必须降级到 `balanced` 或 `hd60`，不能把缺字段当作协议错误。

## 两端实现建议

Android Client：

```kotlin
val hello = JSONObject().apply {
    put("client_name", Build.MODEL ?: "android tablet")
    put("platform", "android")
    put("protocol_version", 0)
    put("supported_codecs", JSONArray().put("h264"))
    put("supported_h264_stream_formats", JSONArray().put("annex_b"))
    put("display_capabilities", displayCapabilities.toJson().getJSONObject("display_capabilities"))
}
```

Mac Host：

```swift
let capabilities = hello["display_capabilities"] as? [String: Any]
let selectedProfile = selectProfile(
    requestedProfile: requestedProfile,
    displayCapabilities: capabilities,
    fallback: .hd60
)
```

要求：

- Android 不等待 Mac 回显 `display_capabilities`。
- Mac 忽略未知字段。
- Mac 解析失败时记录 warning 并降级，不断开连接。
