## 任务目标

在已经跑通 USB 屏幕共享的基础上，把 Mac Host 做成可选择画质/帧率档位的 M1.1 版本。重点是 USB 直连稳定性、画质、帧率和可观测性，不做触控、不做虚拟显示、不做 Wi-Fi。

目标设备是用户手上的小米 Pad。不要硬编码设备原生分辨率或刷新率；Mac 端必须以 Android 运行时上报的 `display_capabilities` 为准。

- 稳定档：1280x800 @ 30fps，8 Mbps。
- 高清档：1920x1200 @ 60fps，16-24 Mbps。
- 设备原生档：由 Android 端读取并上报的当前/推荐 Display Mode 决定，码率按像素量和目标 FPS 动态估算，可失败但必须可诊断。

## 实现要求

1. 在 CLI 与 SwiftUI App 中支持画质档位：
   - `balanced`: 1280x800, 30fps, 8 Mbps
   - `hd60`: 1920x1200, 60fps, 16-24 Mbps
   - `detected-native`: 使用 Android `HELLO.display_capabilities.current_mode`，不要写死分辨率或刷新率
   - `detected-native-safe`: 使用 Android 原生分辨率，但 FPS 上限先限制到 60，码率动态估算
2. 保留手动参数覆盖：
   - width
   - height
   - fps
   - bitrate
   - keyframe interval
3. VideoToolbox 编码参数需要适配 60fps：
   - 禁用 B-frame。
   - 关键帧间隔默认 1 秒或 2 秒，但需要可配置。
   - 失败时降级并给出明确日志。
4. ScreenCaptureKit 采集与缩放策略必须明确：
   - 不要求 Mac 真实屏幕等于 Android 原生分辨率。
   - 输出分辨率以编码输出为准。
   - 如果缩放成本过高，报告中写明瓶颈。
5. SwiftUI App 显示：
   - 当前档位
   - 输出分辨率
   - 目标 FPS / 实际 FPS
   - 目标码率 / 估算实际码率
   - 平均编码耗时
   - 客户端连接状态
   - 最近错误
6. 断开连接后必须停止采集与编码，避免 Mac 空转发热。

## 验证要求

必须至少完成：

```bash
cd src/mac-host
swift build
./.build/debug/machost --dump /tmp/mad-balanced.h264 --profile balanced --dump-duration 5
./.build/debug/machost --dump /tmp/mad-hd60.h264 --profile hd60 --dump-duration 5
```

如果 `detected-native` 无法稳定 dump，不能写“完成”，必须写具体错误、CPU/编码耗时或 VideoToolbox 失败码。

设备原生档验证不能用写死参数冒充。必须注明能力来源：

- 来自 Android `HELLO.display_capabilities`
- 来自临时 JSON 文件
- 未取得能力，降级到 `hd60`

可选验证：

- 与 Android 真机 USB 联调 5 分钟。
- 记录 30fps 与 60fps 下的主观延迟和卡顿。
