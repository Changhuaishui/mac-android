# AGENTS.md

本文件是 mac-android 项目的项目级 AI 工作规范。任何 AI Agent 进入本项目时，必须优先遵守本文件。

## 项目定位

本项目是个人使用的 macOS + Android 副屏系统。

优先级从高到低：

1. 本地可控：无账号、无云服务、无遥测。
2. 安全清晰：屏幕采集、输入注入、网络传输和日志都遵循最小权限。
3. 阶段可验证：每个阶段都有目标、证据、验证和复盘。
4. 工程可演进：先跑通闭环，再逐步增加输入、虚拟显示和体验优化。

不要把本项目做成公网远程桌面、商业化产品、账号系统或云中继服务。

## 工作循环

每轮任务必须按这个顺序推进：

```text
目标定义
→ 任务卡
→ 架构或范围确认
→ 实施
→ 验证
→ 交付记录
→ 当前工作台更新
```

没有任务卡时，不开始大范围代码实现。小的文档修正可以直接做，但必须在最终回复中说明。

## 范围控制

每个任务只能解决任务卡中明确写出的目标。

禁止：

- 因为“顺手”引入新架构。
- 因为“更先进”替换已冻结的阶段技术选型。
- 把 POC 写成最终安全方案。
- 把竞品灵感直接变成当前阶段任务。
- 把未验证内容写成完成。
- 在没有用户授权时安装依赖、连接设备、写系统权限、提交代码或发布。
- 让 Mac 端和 Android 端互相等待才验证。

M1 必须采用解耦验证：Mac 端用 dump client 或文件输出自证，Android 端用 fixture 或 replay server 自证。

需要再次确认：

- 使用 macOS 私有 API。
- 开始 DriverKit 或系统扩展开发。
- 安装 SDK、证书、驱动或全局工具。
- 连接真实 Android 设备并安装 App。
- 引入公网访问、账号体系、远程中继、遥测或云服务。

## 架构决策

重大技术选择必须写入 `设计/ADR/`。

必须写 ADR 的情况：

- 替换核心技术选型。
- 引入新传输协议。
- 引入新平台框架。
- 改变安全边界。
- 进入虚拟显示器路线。
- 引入外部服务或长期依赖。

ADR 至少包含：

```text
决策：
原因：
替代方案：
暂不选择原因：
重新评估条件：
```

## M1 冻结技术选型

M1 目标只验证视频链路：

```text
Mac 采集现有屏幕
→ H.264 编码
→ TCP 发送
→ Android 接收
→ MediaCodec 解码
→ SurfaceView 显示
```

M1 固定选型：

- Mac：Swift。
- 采集：ScreenCaptureKit。
- 编码：VideoToolbox H.264。
- 传输：TCP 长连接。
- Android：Kotlin。
- 解码：MediaCodec。
- 渲染：SurfaceView。
- 协议：自定义 protocol v0。

M1 明确不做：

- macOS 虚拟显示。
- 输入回传。
- 公网访问。
- 账号系统。
- 复杂配对和加密。
- 商业化安装体验。
- WebRTC、QUIC、UDP/RTP、HEVC、Flutter、Electron。

这些方案可以记录为后续评估，但不能阻塞 M1。

## 代码规范

总体要求：

- 代码优先清晰、可调试、可替换。
- 模块边界必须贴合 `Mac Host / Android Client / Protocol`。
- 不为了抽象而抽象。
- 不做无关重构。
- 新增依赖必须说明理由、收益和替代方案。
- 公共协议字段必须有版本号、类型、长度和错误处理。

Swift 代码：

- 优先使用 Apple 原生框架。
- 采集、编码、传输分模块，不混在一个巨大文件里。
- 异步边界必须清楚，失败必须返回可见错误。
- 不静默吞掉 ScreenCaptureKit、VideoToolbox 或网络错误。

Kotlin 代码：

- 解码、渲染、连接状态分模块。
- MediaCodec 状态变化必须可诊断。
- Surface 生命周期必须显式处理。
- 失败时显示可见状态，不允许黑屏无日志。

Protocol 代码：

- 协议结构必须有文档。
- 帧头字段前后一致。
- 大小端、时间戳单位、flags 含义必须固定。
- 任何协议变更都要更新文档和测试样例。

## 安全与隐私

任何阶段都必须遵守：

- 不上传屏幕内容到云端。
- 不保存屏幕画面。
- 不记录账号、密钥、完整输入文本。
- 日志只记录连接状态、错误码、帧率、码率、延迟和必要诊断信息。
- 断开连接后停止采集或进入明确暂停状态。
- 默认只监听本机或局域网。
- 不绕过 macOS 权限提示。

如果安全与调试便利冲突，先保证安全边界，再设计调试方式。

## 验证要求

每次代码完成后必须创建或更新 `交付记录/*.md`。

验证结果只允许写：

- 通过
- 失败
- 未验证
- 不适用

不能写“应该可以”“看起来没问题”来代替验证。

M1 最小验收：

- Android 能显示 Mac 画面。
- 720p 30fps 作为第一目标。
- 连续运行 10 分钟。
- 有 FPS、码率或延迟的基础日志。
- 断线或失败有可见错误。

如果没有真实 Android 设备或权限，必须明确写“未验证”和原因。

M1 进入端到端联调前，必须先满足：

- `设计/protocol-v0.md` 完成。
- Mac 端可以被 dump client 或等价假客户端验证。
- Android 端可以播放 H.264 fixture 或 replay server 流。
- 双方都明确 H.264 是 Annex B 还是 AVCC。

## 多 Agent 协作

一个任务只有一个主 Agent。

默认分工：

- Codex：主 Agent，负责架构、代码、文档、验证交付。
- Kimi：辅助 Agent，负责真实浏览器、资料查证、页面或体验观察。
- Claude：辅助 Agent，负责只读复核、风险检查、第二意见。

辅助 Agent 默认只读，不直接修改文件。

辅助 Agent 输出必须使用：

```text
检查对象：
发现的问题：
证据：
影响：
建议：
未检查内容：
```

辅助 Agent 的建议不是最终结论，必须由主 Agent 或用户合并。

## 文档更新规则

完成任务时至少检查是否需要更新：

- `当前工作台.md`
- 当前 `任务记录/*.md`
- 当前 `交付记录/*.md`
- `设计/ADR/`
- `设计/技术选型-M1.md`
- `参考资料/竞品灵感池.md`

长期规则只在重复出现、风险较高、可明确表达、可检查时加入本文件。

不要因为一次性偏好就扩写 AGENTS.md。

## 完成定义

一个任务完成前，必须能回答：

- 本轮目标是否完成？
- 改了哪些文件？
- 为什么这些改动属于本轮范围？
- 执行了哪些验证？
- 哪些内容未验证，原因是什么？
- 是否改变了架构、安全或依赖边界？
- 下一步是什么？

最终回复要简短说明结果、关键文件和验证状态。
