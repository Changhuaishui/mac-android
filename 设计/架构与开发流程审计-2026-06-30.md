# 架构与开发流程审计

日期：2026-06-30

## 审计结论

当前项目架构方向正确：先跑通视频闭环，再做输入回传，最后攻克 macOS 虚拟显示。这个顺序能避免第一阶段被最高风险的虚拟显示器拖住。

当前开发流程也基本可用：已有当前工作台、任务卡、验证交付、Loop 工程和 Agent 边界。但在进入 M1 代码前，需要补两个机制：

1. 架构决策记录 ADR，用于冻结阶段性技术选择。
2. 竞品灵感池，用于保存 UU 远程等产品启发，但不让灵感直接改变当前任务范围。

## 当前架构评价

### 优点

- Mac Host、Android Client、Protocol 三层边界清楚。
- M1 使用现有屏幕或窗口，避开虚拟显示这个最高风险点。
- ScreenCaptureKit、VideoToolbox、MediaCodec、SurfaceView 的组合符合原生低延迟路线。
- TCP 作为 M1 传输方案足够简单，适合先跑通闭环。
- 安全边界已经提前写入：不接公网、不上传云端、不保存画面、断连停采集。

### 缺口

- 没有 ADR，后续 Agent 容易重新争论 WebRTC、Flutter、Electron、HEVC、虚拟显示等问题。
- 没有竞品参考沉淀区，UU 远程这类灵感容易散落在聊天中。
- M1 还缺明确任务拆分：Mac 采集编码、Android 解码渲染、协议、验证、风险复核。
- 协议草案需要在 M1 前进一步落成可实现的二进制帧格式。

## 开发流程评价

### 保留

- 一个任务只有一个主 Agent。
- 辅助 Agent 默认只读。
- 所有任务从当前工作台和任务卡恢复上下文。
- 已验证、未验证、不适用必须分开写。
- 失败进入下一轮，而不是靠聊天记忆消化。

### 增补

- 每个阶段开始前增加一次“架构冻结”。
- 关键技术选择必须有 ADR。
- 竞品想法先进入灵感池，只有经过任务卡确认才进入架构。
- M1 开发任务按可独立验证的模块拆分。

## M1 推荐开发流

```text
M1 架构冻结
  ↓
ADR 确认
  ↓
协议最小帧格式
  ↓
Mac 采集编码 POC
  ↓
Android 解码渲染 POC
  ↓
端到端联调
  ↓
验证交付
  ↓
复盘与 M2 输入链路准备
```

## Kimi 约定

Kimi 的角色是辅助调研和体验观察，不承担架构最终决策。

推荐提示：

```text
你是 mac-android 项目的辅助 Agent，不是主 Agent。
本任务主 Agent 是 Codex。
你只做只读调研、真实环境观察和交付清单检查，不修改文件，不改变架构决策。

请按以下格式输出：
检查对象：
发现的问题：
证据：
影响：
建议：
未检查内容：
```

## UU 远程灵感处理

UU 远程可作为竞品体验参考，但当前不改变 M1 技术路线。

可参考方向：

- 连接引导。
- 设备发现。
- 弱网提示。
- 远程控制输入体验。
- 码率、清晰度、帧率选项。
- 诊断信息展示。

暂不参考方向：

- 公网中继。
- 账号体系。
- 商业化会员功能。
- 云端设备列表。
- 任何不透明遥测。
