# 技术选型：M1 视频链路 POC

日期：2026-06-30

## M1 目标

M1 只验证一条最小视频链路：

```text
Mac 采集现有屏幕
→ H.264 编码
→ TCP 发送
→ Android 接收
→ MediaCodec 解码
→ SurfaceView 显示
```

不做：

- 虚拟显示。
- 输入回传。
- 账号系统。
- 公网访问。
- 复杂加密和配对。
- 商业化安装体验。

## 推荐选型

| 模块 | 选择 | 理由 |
|---|---|---|
| Mac 语言 | Swift | 原生调用 ScreenCaptureKit、VideoToolbox、Network.framework 最顺 |
| Mac 形态 | 先 CLI，后菜单栏 App | CLI 更适合验证链路，菜单栏后补 |
| 屏幕采集 | ScreenCaptureKit | macOS 官方现代屏幕采集框架 |
| 编码 | VideoToolbox H.264 | 硬件编码、Android 兼容性好 |
| 传输 | TCP 长连接 | 实现简单，USB/局域网下足够做 POC |
| Android 语言 | Kotlin | 原生 Android 开发主线 |
| Android UI | 单 Activity + SurfaceView | 最少 UI，直接验证解码渲染 |
| Android 解码 | MediaCodec | 原生硬件解码，能直接输出到 Surface |
| 协议 | 自定义 protocol v0 | 长度前缀 + 固定帧头，方便调试 |
| 构建 | Xcode/SwiftPM + Gradle | 分别贴近平台原生工具链 |

## 暂不选择

| 方案 | 暂不选择原因 | 何时重新评估 |
|---|---|---|
| WebRTC | POC 前期依赖和调试成本高 | Wi-Fi 弱网和 NAT 需求明确后 |
| QUIC/UDP/RTP | 传输复杂度高于当前阶段收益 | TCP 延迟或丢包问题成为瓶颈后 |
| HEVC | 兼容性和调试成本高于 H.264 | H.264 画质/带宽不可接受后 |
| Flutter | 对底层 Surface、MediaCodec、权限调试不如原生直接 | 后续需要跨端 UI 时 |
| Electron | 不适合 Mac 底层采集编码和权限控制 | 不建议用于核心 Host |
| Rust/C++ 首发 | 调用 Apple/Android 原生 API 成本更高 | 性能瓶颈明确后再局部引入 |
| 虚拟显示 | 是 M3 核心风险，不应阻塞 M1 | 视频链路和输入链路稳定后 |

## 协议 v0 建议

使用单 TCP 连接先跑通，消息结构：

```text
u32 magic        // 'MADS'
u16 version      // 0
u16 type         // hello / video_config / video_frame / ping / error
u64 sequence
u64 timestamp_ns
u32 flags
u32 payload_len
payload
```

M1 只需要：

- `HELLO`
- `VIDEO_CONFIG`
- `VIDEO_FRAME`
- `PING`
- `ERROR`

M2 再增加：

- `INPUT_EVENT`
- `KEY_EVENT`
- `CURSOR_EVENT`

## H.264 格式约定

M1 必须尽早确认 Mac 输出给 Android 的 H.264 格式：

- 优先采用 Annex B byte stream，帧内带 start code。
- `SPS/PPS` 在连接开始和关键帧前发送。
- 每个 `VIDEO_FRAME` 标明是否 keyframe。
- Android 解码失败时可以请求重新发送配置和关键帧。

如果 VideoToolbox 默认输出 AVCC，需要在 Mac 端转换为 Annex B，或者 Android 端正确配置 csd-0/csd-1。

## 性能指标

M1 最小指标：

- 720p。
- 30fps。
- 连续运行 10 分钟。
- Android 可见画面。
- 平均端到端延迟有粗略估计。
- 失败可见，不静默卡死。

可选指标：

- 1080p 30fps。
- 1080p 60fps。
- 码率 8-20 Mbps 手动可调。

## 安全底线

M1 即使是 POC，也必须遵守：

- 默认只监听本机或局域网。
- 不接公网。
- 不保存画面。
- 日志不记录屏幕内容和完整输入文本。
- 断开连接后停止采集。
