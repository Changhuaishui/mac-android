# M1 解耦验证策略

日期：2026-06-30

## 问题

M1 如果让 Mac 端和 Android 端直接互相等待，会出现死锁：

- Mac 端采集、编码、TCP 发送遇到问题时，无法判断是自己问题还是 Android 端未准备好。
- Android 端等待 Android Studio、Gradle、模拟器或设备调试时，无法验证 Mac 端是否正确。
- 两边都缺少固定输入和固定输出，导致互相阻塞。

## 原则

M1 必须改成“契约优先 + 假对端 + 固定测试资产”。

任何实现 Agent 都不能把“对端没完成”作为唯一阻塞理由。每一端都必须能用假对端或 fixture 自证当前进度。

## 解耦件

| 解耦件 | 作用 | 负责人 | 阶段 |
|---|---|---|---|
| `protocol-v0.md` | 固定 Mac/Android 消息格式 | Protocol Agent | 最先完成 |
| H.264 fixture | Android 无需等待 Mac 也能解码 | Protocol Agent / Android Agent | M1-A |
| replay server | 模拟 Mac Host 向 Android 发帧 | Protocol Agent | M1-A |
| dump client | 模拟 Android Client 接收 Mac 帧 | Protocol Agent / Mac Agent | M1-A |
| Mac capture-to-file | Mac 不等 Android，先证明编码输出 | Mac Agent | M1-B |
| Android asset playback | Android 不等 Mac，先证明解码渲染 | Android Agent | M1-C |

## 并行推进方式

```text
Protocol Agent
  ├─ protocol-v0.md
  ├─ H.264 fixture 约定
  ├─ replay server 约定
  └─ dump client 约定

Mac Agent
  ├─ capture -> encode -> write sample.h264
  ├─ tcp server -> dump client 验证
  └─ 不等待 Android App

Android Agent
  ├─ asset sample.h264 -> MediaCodec -> SurfaceView
  ├─ replay server -> TCP -> MediaCodec -> SurfaceView
  └─ 不等待 Mac Host
```

## Mac 端独立验收

Mac 端不需要 Android 也能完成这些验证：

1. 能列出 display。
2. 能采集主屏。
3. 能编码 H.264。
4. 能写出 `sample-annexb.h264` 或明确记录 AVCC。
5. 能启动 TCP server。
6. dump client 连接后能收到持续帧。
7. 连接断开后不无限堆积帧。

Mac 端交付物：

- `agents/reports/M1-MacHost-Agent-报告.md`
- H.264 输出格式说明。
- 运行命令。
- 如果无法采集屏幕，记录具体权限或 API 错误。

## Android 端独立验收

Android 端不需要 Mac 也能完成这些验证：

1. App 能启动。
2. SurfaceView 生命周期明确。
3. 能从 assets 或测试文件读取 H.264 fixture。
4. 能用 MediaCodec 解码并渲染。
5. 能连接 replay server。
6. 能解析 protocol v0 frame。
7. 解码失败时显示错误，不静默黑屏。

Android 端交付物：

- `agents/reports/M1-AndroidClient-Agent-报告.md`
- 构建方式。
- 运行方式。
- 小米平板 6 Pro 验证状态。
- 如果 Android Studio、Gradle、模拟器卡住，必须写明卡点，不得等待。

## Android Studio / 模拟器卡点处理

M1 不依赖 Android 模拟器。

优先级：

1. 真实小米平板 6 Pro。
2. Gradle CLI `assembleDebug`。
3. Android Studio 打开工程。
4. 模拟器。

如果 Gradle 或 Android Studio 下载依赖卡住，Android Agent 仍然要交付：

- 工程结构。
- 关键 Kotlin 文件。
- AndroidManifest。
- MediaCodec 解码逻辑。
- SurfaceView 生命周期逻辑。
- 未编译原因。
- 下一步需要用户授权或网络环境。

不要因为模拟器没下载完成而停工。

## Protocol Agent 优先交付

Protocol Agent 应先完成：

- `设计/protocol-v0.md`
- `测试资产/M1-解耦验证/README.md`
- replay server 的接口约定
- dump client 的接口约定

可先不写完整工具代码，但必须让 Mac/Android 两端知道：

- H.264 是 Annex B 还是 AVCC。
- `VIDEO_CONFIG` 和 `VIDEO_FRAME` 怎么发。
- flags 里的 keyframe 和 config 如何标记。
- 大小端。

## 端到端联调前置条件

只有当以下条件满足，才进入真正端到端：

- Mac 能被 dump client 验证。
- Android 能播放 fixture。
- Android 能连接 replay server。
- protocol v0 文档完成。

否则继续做独立验证，不强行联调。

## 完成定义

M1 阶段不以“一次完整联调成功”作为唯一进度判断。

分层完成：

| 层级 | 完成标志 |
|---|---|
| M1-A | protocol v0 + fixture/replay/dump 约定完成 |
| M1-B | Mac capture/encode/TCP 可被 dump client 验证 |
| M1-C | Android fixture/replay 解码渲染可验证 |
| M1-D | Mac Host + Android Client 端到端显示 |
