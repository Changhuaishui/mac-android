# Loop 工程

## 定义

本项目采用 Loop 工程方式：不是让 AI 全自动接管工程，而是在指定任务范围内，通过明确目标、执行、验证、反馈和规则升级，让项目不断纠错并逼近阶段最优。

参考材料：

- 菜鸟教程《Loop Engineering》：`https://www.runoob.com/ai-agent/loop-engineering.html`
- 本地参考：`/Users/chenji/Downloads/zj/协作优化/Harness工程`

## 基本循环

```text
目标定义
  ↓
计划与边界
  ↓
执行
  ↓
验证
  ↓
复盘
  ↓
规则或任务更新
  ↓
下一轮
```

## 每轮必须回答的问题

### 目标定义

- 本轮只解决什么？
- 明确不解决什么？
- 完成后用户能看到什么？
- 哪些内容必须保留为未验证？

### 计划与边界

- 主 Agent 是谁？
- 辅助 Agent 是否只读？
- 哪些操作允许？
- 哪些操作需要再次确认？

### 执行

- 修改是否紧贴本轮目标？
- 是否引入了无关重构？
- 是否改变了安全边界？

### 验证

- 已执行哪些验证？
- 证据是什么？
- 哪些验证未执行，原因是什么？
- 失败是否已经进入下一轮目标？

### 复盘

- 本轮错误来自需求理解、技术判断、实现、验证还是协作？
- 是一次性问题，还是会重复出现？
- 是否需要升级为长期规则？

## 阶段最优

阶段最优不是全局完美，而是在当前阶段约束下达到最合理结果。

例如：

- M1 的阶段最优是视频链路稳定，不是虚拟显示完美。
- M2 的阶段最优是输入闭环正确，不是所有手势都支持。
- M3 的阶段最优是真扩展屏可用，不是商业级驱动发行。

## 全局最优

全局最优通过多个阶段逐步逼近：

1. 保留正确的架构边界。
2. 不让短期 POC 污染长期安全设计。
3. 把失败转成可复用的验证项。
4. 只将反复出现的问题升级为规则。

## 规则升级条件

一条经验成为长期规则前，至少满足一项：

- 同类错误重复出现。
- 一旦遗漏会造成高风险。
- 能被一句短规则表达。
- 能被检查或验证。

不能升级为长期规则的情况：

- 只是个人偏好。
- 只发生一次。
- 依赖具体上下文。
- 无法验证。

## 排除异化

本项目不接受以下异化：

- 为了“自动化”而自动化。
- Agent 自己扩大目标。
- 用大量文档替代验证。
- 把未验证写成已完成。
- 辅助 Agent 直接接管主任务。
- 为了追求全局最优而拖死当前阶段。

## 共通协助

多 Agent 共享：

- 同一份当前工作台。
- 同一份任务卡。
- 同一份完成定义。
- 同一份验证交付。
- 同一套术语和状态。

辅助 Agent 的价值是提供证据、检查遗漏、复核方案，而不是制造新的任务中心。
